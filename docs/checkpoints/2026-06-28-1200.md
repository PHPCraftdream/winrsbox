vs read-miss use different case → two physical paths → git confused") is
under-specified: git object paths are lowercase-hex so case-preserve is a
no-op there; NTFS is case-insensitive so `mirror(lower)` and
`mirror(original)` open the same file. Real candidate mechanism (INFERRED)
is git's `index-pack` tmp_pack_XXXXXX rename cycle, but the agent's trace
showed tmp_pack create+rename completed normally. Mechanism still
unproven — agent took the discipline of stopping rather than shipping a
guess. I explicitly recommended a **strategic redirect**: instead of

lowercase (no regression risk by construction), original case is returned
through the `NtQueryDirectoryFile` hook (which is already installed and
verified working — `.winrsbox` hiding still works in `dir`). This routes
around the unknown regression instead of debugging through it.

  physical storage stays lowercase) **over** retrying physical-case-preserve
