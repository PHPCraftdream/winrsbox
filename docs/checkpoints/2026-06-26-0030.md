## Session summary

This session hardened **winrsbox** — a Windows filesystem sandbox for AI agents
(`D:\dev\rust\fs-sandbox\`, repo `github.com/PHPCraftdream/winrsbox`) — driven by the standing
goal "довести песочницу для работы агентов. Работу поручать /crush" (bring the sandbox ready for
agent work; delegate work to /crush). Nearly all implementation was delegated to **crush**
sub-agents; the orchestrator only designed specs and zero-trust-verified every result by
re-running tests/E2E on freshly built binaries.

We fixed a long chain of real bugs, each surfaced by automated E2E harnesses (`repro/e2e/`,
`repro/e2e2/`), in roughly this order: (1) IPC self-terminate cascade under MSYS2 first-run
(recover pipe config via a `Local\WinRsBoxSession` shared section when env is scrubbed); (2)
FS-isolation — out-of-project writes must go into the CoW overlay, not the real disk; (3)
whiteout deletes (`Mode::Hidden` + WHITEOUTS table); (4) overlay path-leak masking via
NtQueryInformationFile; (5) git-under-sandbox: removed an unconditional `SetCurrentDirectoryW`
override that forced every child's CWD to project_root (this was the user's *original* "repo
created in the wrong folder" bug), rename-to-overlay redirect, key normalization, directory-enum
fix; (6) nested-sandbox blocking (transparent delegation, argv preserved); (7) the FINAL git
blocker — `git add/commit` failing "unknown error reading configuration files". A clean
`GIT_TRACE` diagnosis (NOT guessing) found the real root cause: a **self-block** — git does a
relative-open of `.git/config` from a CoW overlay handle; our `resolve_handle_path` resolved it
to the `.winrsbox\…` overlay path, and our own `.winrsbox` denylist guard blocked it. Fix was a
few lines: `unmirror_overlay_handle_relative` (overlay→virtual before denylist/decide) in
`hooks.rs`. No bindflt/driver needed — an earlier "architectural conflict / need bindflt"
contemplation turned out to be over-engineering once diagnosis replaced guessing.

After that: committed everything (`d326ca2` and predecessors), then did GitHub polish — repo
topics (17), description, a standard badge row in README. Finally migrated dependency **ktav
0.3.1 → 0.6.1** and rewrote the `.ktav` docs for the 0.6.1 format. User's authoritative hints
("comments are `##`, type-hints removed") were confirmed empirically against the real parser
(crush initially mis-guessed `#`). All committed and pushed to `origin/master` (HEAD `e2ed63e`).

A notable incident: `crush sessions kill` failed with PID=0 (lock held by OS); a stale
`winrsbox-git-final` session kept running in parallel, overwriting a revert and corrupting the
tree (franken-mix). Resolved by finding real PIDs via PowerShell `Get-CimInstance` and
`taskkill //F //T //PID`, then `git checkout` to clean `b1cba57` and relaunching one clean
session. Lesson recorded: kill by real PID, never let crush sessions accumulate.

Recurring footgun: `scripts/build-deploy.sh` once falsely reported "unchanged" and shipped a
stale binary (false test failures); crush fixed it to resolve target dir via `cargo metadata`.
Environment uses `CARGO_TARGET_DIR=D:\dev\rust\.cargo-target`, so the reliable deploy is:
`cargo build -p winrsbox -p hook --release` then manually `cp` the two artifacts into
`D:\dev\rust\fs-sandbox\bin\`. Run anything under the sandbox with
`winrsbox.exe --cwd <project_root> [--trace] -- <prog> [args]`; when invoking git via git-bash
use `MSYS_NO_PATHCONV=1 MSYS2_ARG_CONV_EXCL='*'` so `/c` isn't mangled to `C:/`.

Current verified state: E2E v1 19/19 PASS, E2E v2 27/27+17/17 PASS (node/npm/python/git-advanced/
files/whiteout/isolation/nested/enum/passthrough). The sandbox is engineering-complete for agent
